justhtml.sh — comment overlay — design iteration

Four side-by-side takes on the Phase-2 comment overlay. Each renders a faked doc inside the real man-page viewer chrome (the overlay lives in the shell over a sandboxed iframe in production). All variants implement every interaction: hover-to-open thread, click-to-pin, reply, selection toolbar, add-comment, emoji reaction, plus a resolved thread and an orphaned comment. Frames are scaled down — use open full screen to interact properly.

A — margin bubbles (literal spec) avatar bubbles in the right margin; hover pops a modal beside the bubble; vertical icon toolbar on selection. open full screen ↗
B — google-docs rail bubbles collapse into a persistent right rail; cards align to highlights and expand in place to reply. open full screen ↗
C — man-page footnotes no avatars, no fill: anchors get superscript [1][2] markers; monospace footnote cards; text-button toolbar. open full screen ↗
D — synthesis (quiet marginalia) doc looks untouched at rest (dotted underline + margin dots); highlight is a transient hover state that blooms a docs-capable card. open full screen ↗

Design tension explored

  • A vs B — floating margin bubbles (low footprint, but stacking/collision problems when comments cluster) vs a fixed rail (predictable layout, costs horizontal space, less "marginalia").
  • C — pushes the brand's man-page restraint to its limit: comments read as footnotes, never compete with the prose. Risk: avatars/identity are deemphasized.
  • D — argues the heaviest Google-Docs element is the permanent yellow fill; making it a hover-only state lets a full comment layer coexist with a light man-page doc.

Judgments made beyond the spec

  • Overlapping highlights / clustered bubbles: A and D push colliding margin markers down by a min-gap; B aligns rail cards to highlight tops but never lets two cards overlap (each card's top is clamped below the previous card's bottom — the Google-Docs algorithm). C sidesteps it entirely with thin text rows.
  • Orphaned comments: A/D show a hollow/dashed margin marker with no in-text highlight; B shows a dashed card in the rail under the anchored ones; C lists them in a dedicated ORPHANED section at the doc's end — the most honest footnote treatment.
  • Resolved threads: dimmed + collapsed everywhere; B/C gate them behind a show-resolved toggle (matches the all-threads-view spec); the highlight degrades to a faint dashed underline rather than a fill.
  • Zero comments: every variant renders the bare doc + chrome with no gutter additions (toggle comments off to see it).
  • Cross-element anchors: anchoring is against rendered text content (W3C text-quote), with a whitespace-normalized fallback search, so a quote spanning element boundaries still resolves; if DOM-wrapping the span fails, the bubble/dot still appears (anchor known, highlight skipped).
  • Mobile: not a primary target (agent-authored docs are read on desktop), but the rail (B) would become a bottom sheet and margin markers (A/C/D) would dock to the right edge; the comparison grid collapses to one column under 920px.
  • Hover-intent: all hover-to-open variants use a ~120ms open delay, a ~160ms close delay, and an invisible hover-bridge element between marker and card so the cursor can cross the gap without flicker.
